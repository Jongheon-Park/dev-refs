## PROJECT DISCOVERY

**Trigger**: `1.AI_ARCHITECTURE.md` has `[TBD]` content, OR `3.AI_DEVLOG.md` Phase 1 is `⏸️ Pending`.

This is a **cold start** — the AI must analyze the project before any coding tasks.

### Discovery Steps

1. **Scan project root** — identify solution/project files (`.sln`, `.csproj`, `CMakeLists.txt`,
   `package.json`, `Cargo.toml`, etc.) and determine the tech stack.
2. **Map directory structure** — walk top-level directories, identify each module's role.
3. **Identify key files** — entry points, core modules, shared utilities, config files.
4. **Check for existing context** — read any existing `CHANGELOG.md`, `README.md`, or architecture docs.
5. **Fill templates**:
   - Populate `1.AI_ARCHITECTURE.md` with real system overview, layers, modules, dependencies.
   - Populate `2.AI_PATH_NAV.md` with actual directory paths and module mapping.
6. **Log in devlog** — mark Phase 0 (Environment Setup) and Phase 1 (Project Discovery) as ✅ Completed
   in `3.AI_DEVLOG.md`.
7. **Confirm with user** — present a brief summary of what was discovered and ask:
   "Project discovery complete. Does this look accurate? Shall I proceed?"

4. **Risks**: Breaking changes, compatibility

**AI must ask:** "Do you approve this design to proceed?"

## IMPLEMENTATION PROCESS

1. **Clarify & plan.** State assumptions and define success criteria before starting. Break work into
   steps — implementation, docs, and commit. Check them off as you progress.
2. **Implement & test.** Write and run unit tests alongside non-trivial pure logic. When tests fail,
   diagnose whether the **test assumptions** or the **implementation logic** is wrong — don't blindly
   fix the implementation to match a bad test. For UI/visual features, request manual testing instead.
3. **Verify.** Run the full test suite and integration test harness. Run any build or asset-import
   step after editing relevant files. All checks must pass before committing.
4. **Update docs.** After non-trivial tasks:
   - Update `3.AI_DEVLOG.md` if project state or phase changed.
   - Update `# MEMORY` section at the bottom of this file (what was done, what's next).
   - If completing a ticket phase, append `### Result` to that phase in the ticket doc.
   - Prune aggressively: keep docs focused on current state.
5. **Commit.** Create git commits broken down by logical units. Include an **AI Context** section:

   ```
   <type>(<scope>): <summary>

   <what changed — brief>

   ## AI Context
   - <decision rationale, rejected alternatives, user directives, trade-offs>

   Co-authored-by: Copilot <223556219+Copilot@users.noreply.github.com>
   ```

## CODE STANDARDS

1. **Simplicity.** Write the simplest code that works. Implement fully when the spec is clear —
   judge scope by AI effort, not human-hours.
2. **Surgical changes.** Change only what the task requires. Follow existing style. Every changed
   line must trace to the request.
3. **Module structure.** Split files at ~300 lines. Extract an entry file containing doc comments
   and public re-exports only — reading it alone conveys the module's interface.
4. **Hot-path performance.** In performance-critical paths, prefer minimal allocation and data
   locality over convenience abstractions. Apply only when benefit clearly outweighs complexity cost.

## FUNCTION ANALYSIS PROTOCOL

### Phase 1: LOCATE
- Search: `symbol:FunctionName`
- Report all matches (file:line)

### Phase 2: READ
- Extract signature, parameters, return type
- Quote code (5-20 lines)

### Phase 3: TRACE (Minimum 3 levels)
- List all called functions
- Search each implementation
- Build call chain

### Phase 4: VERIFY
- Present findings
- Ask: "Analyze deeper?"

**Stop only at:** stdlib, OS API, or user says "stop"

## MFC THREADING RULES

- **NO direct UI access from worker threads**
- Use `PostMessage()` (async) or `SendMessage()` (sync)
- Protect shared resources with `CCriticalSection` or `std::mutex`

## DEPENDENCY API NOTES

- **`refs/deps/<package>.md`** stores verified API facts for libraries whose actual API differs
  from training knowledge or is too new to be known.
- **When to read:** Before writing code that uses a package listed in `# MEMORY → Documented
  Dependencies`. Also check on compile/type errors that look like wrong signatures or changed fields.
- **When to write/update:** After discovering API drift (wrong arg count, renamed types, removed
  methods) or after learning a previously-unknown package's API.
